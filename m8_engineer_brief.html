M8 · technical brief (for an engineer)

M8 — a deterministic-first personal AI agent

M8 is a web-based AI assistant with two jobs: (A) run a real business — a Bolt ride-hailing fleet in Riyadh (P&L, driver pace, alerts, reports), and (B) act as a research engine that attacks small, unsolved math problems with machine-checked proofs. The whole design bet is that an LLM should orchestrate and narrate, but never own the truth — deterministic code computes facts, the model only explains them. Live at m8-alpha.vercel.app.

The stack (all on a mostly-free tier)

Frontend vanilla JS

Single-page chat PWA — index.html + js/{app,chat,voice}.js. Streaming (SSE), file/image attach, push-to-talk voice. No framework.

Backend Vercel

Serverless Node functions in api/*.js (Hobby plan, 12-function cap). Cron jobs for the nightly research loop + the 6am fleet brief.

LLM Gemini

Google gemini-2.0-flash primary, with Groq/Mistral fallbacks on quota. Used for orchestration + narration only — never as the source of a number or a proof.

Database Supabase

Postgres + pgvector. Stores fleet data, the research memory graph, the priority ledger, provenance/trust tiers, and run telemetry.

Formal proofs Cloud Run

m8-lean-check — a Lean 4 + Mathlib service exposing /check. It is the only thing allowed to mark a math claim "proven".

Delivery Resend

Transactional email (the morning fleet brief). The first "reach the user outside chat" seam; nudges/WhatsApp drafts build on it.

Request lifecycle — what happens on one chat message

1 · Browser → api/chat.js / api/chat-stream.js
Message + history + any attachments POSTed; the stream path replies over SSE.
2 · Orchestrator hard-route spine — lib/orchestrator.js
Regex-first, first match wins. A deterministic ladder of detectors (fleet query, morning brief, command-center priority, research command, …) decides if a request is handled by code before any LLM call. This is the deterministic-first core.
3 · Tool-decision router — lib/router.js / intentClassifier.js
When no hard route fires, classify intent: answer · search (live web) · compute (code-exec) · clarify. Source-trust + recency gates rank any web results before they reach the model.
4 · Deterministic engines compute the facts — lib/*
Fleet P&L, finance, the priority engine, the math probes, etc. run as pure code and return a structured "packet" of ground-truth numbers/results.
fleet.jsfinance.jscommand-center.jsmorning-brief.jslemma-dag.jsmemory-graph.js
5 · LLM narrates the packet — under honesty directives
Gemini turns the packet into prose. It is told it may not invent numbers, may not call "tested" the same as "proven", and must hedge on weak sources. An Odysseus red-team battery of permanent probes guards against overclaim regressions.
6 · Persist + stream back
Memory/provenance written to Supabase; reply streamed to the browser. Every write fails safe — a DB error degrades, it never fabricates.

The one design principle everything hangs on

Code computes truth; the model only narrates it. A number comes from a deterministic engine, not the LLM. A math claim is "proven" only when the Lean checker says so — never because the model is confident. Memory carries explicit provenance + trust tiers (verified / empirical / heuristic / unverified / refuted), and the read path surfaces the most-trusted first. The product's headline rule: it never claims more than it can back up.

The two tracks

Track A — Operations (the business today)

✓
Verified fleet P&L, multi-company registry, KSA labour/EOSB/ZATCA playbooks, deck/Excel/PPTX export.
✓
Daily 5,000-SAR-pace "morning brief" (the SAR target is a Bolt bonus tier), Arabic driver-nudge drafts, anomaly alerts, "what changed & why" analysis.
◑
True profit / driver cost profiles — not yet modelled (rental + Bolt bonus economics need capturing first).

Track B — Research (unsolved-problem engine)

✓
Idea → testable claim → exhaustive check → "observed through N" (never "proven"). Two problem families (Collatz, reverse-and-add).
✓
Human-gated lemma-DAG decomposition; leaves discharged through the Lean checker; a nightly autonomous loop (L5).
◑
The L5 promotion gate sits at 0/3 clean nights (honesty graders, now fixed); no open problem cracked — and by design it never claims one is.

Honest current state — what's solid vs half-built

  • Solid & live: the chat app, the deterministic orchestration spine, fleet/finance engines, the priority "Command Center", the honesty layer + Odysseus battery, Lean verification of small leaves.
  • Built but unproven: the autonomous nightly loop (capability live, promotion gate not yet banked).
  • Known gap (just found): the document-ingestion pipeline (PDF/EPUB OCR → knowledge graph) exists in code but has 0 books actually persisted — prior ingests died on serverless timeouts. Needs a resumable/chunked ingest before it's real. A read-only inventory endpoint (GET /api/knowledge-inventory) now reports the true state.
  • Deferred: native mobile/PWA, calendar/email/task integrations, multi-platform order ingestion (only Bolt is deep today).
One-line summary for a busy engineer: a Vercel + Supabase + Gemini chat agent whose defining choice is a deterministic hard-route spine in front of the LLM, so code (and a Lean proof checker) owns every fact and the model only explains — built to run a real fleet business and to probe small math conjectures without ever overclaiming.
